# Notes 08: Binary Builds and Registry Workflow

## Mission Focus

### BuildConfig (OpenShift)

- Declarative build configuration.
- Source types: Git (S2I), Binary, or Dockerfile.
- Binary mode accepts pre-built artifacts instead of source code.

### Build (OpenShift)

- Single build run instance created from BuildConfig.
- Owned by BuildConfig, tracked in etcd.
- Build pod executes on cluster.

### Build Pod (Kubernetes Runtime)

- Temporary pod that executes build strategy.
- Runs as `builder` ServiceAccount.
- Has credentials to push to internal registry.

## OpenShift Layer

### ImageStream

- Tracks images built by BuildConfig.
- Maintains tag → digest mapping.
- Controllers watch for new images and update tags.

### Internal Image Registry

- OpenShift-native registry (not external DockerHub).
- Integrated with API Server and RBAC.
- ServiceAccounts have automatic pull credentials.

## Kubernetes Layer

### Deployment

- Consumes images from ImageStream or direct registry reference.
- Scheduler assigns pods to nodes.
- Kubelet pulls image from registry.

### Service (optional)

- Exposes deployment endpoints.

## Binary Build Flow

1. **User** creates BuildConfig with `source.type: Binary`.
2. **User** runs `oc start-build --from-dir` or `--from-file`.
3. **API Server** creates Build object.
4. **Controller** creates build pod on a node.
5. **Build Pod** extracts artifact, executes Dockerfile.
6. **Docker step** builds image and pushes to internal registry.
7. **API Server** receives push notification.
8. **ImageStream Controller** updates ImageStream tags with digest.
9. **Deployment** can now pull image by tag or digest.

## Binary vs S2I

| Aspect         | Binary                             | S2I                          |
| -------------- | ---------------------------------- | ---------------------------- |
| Source         | Pre-built artifact                 | Source code                  |
| Build location | Artifact uploaded to cluster       | Code cloned on cluster       |
| Speed          | Faster (no compilation on cluster) | Slower (includes build time) |
| Use case       | Pre-compiled, external build       | Source code on Git           |
| Dockerfile     | User-provided                      | Auto-generated by builder    |

## Registry Authentication

- Build pod runs as `builder` ServiceAccount.
- OpenShift automatically injects registry credentials.
- Deployment pods pull images using default ServiceAccount.
- Default ServiceAccount has `imagepull` role for internal registry.

## Relationship Summary

- BuildConfig → Build pod → Binary artifact + Dockerfile → Image push.
- Build pod → internal registry via service account credentials.
- Image registry → ImageStream controller updates tags and digests.
- Deployment → ImageStream tag or direct registry reference for image pull.
- API Server, etcd, and controllers orchestrate entire workflow.
